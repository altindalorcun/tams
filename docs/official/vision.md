**TAMS — Transcript Analysis and Graduation Control System**

**Vision**

# Introduction

This project delivers a system that determines students' graduation eligibility by analyzing their uploaded PDF transcripts against predefined graduation requirements. Graduation rules are organized into course categories — for example, In-Department Mandatory, Out-of-Department, Technical Elective, and Non-Technical Elective — each with associated course codes, credits, and ECTS thresholds. The system operates based on three primary user roles: Admin, Teacher, and Student.

Within the scope of the project:

- **Admins** define, update, delete, and view graduation rules and conditions (departments, course catalogs, graduation categories, curriculum equivalence rules, and user accounts) within the system.
- **Teachers** upload students' PDF transcripts, trigger automated analysis, and view the results. They can also browse and search the graduation data of students they have previously analyzed.
- **Students** view their own analysis results in read-only mode, provided their transcript has been uploaded by a teacher.

Personal data contained within transcripts (e.g., national identity number) is protected in accordance with data privacy standards. PDF files are transmitted securely over the network, processed entirely in temporary memory, and are not stored permanently on the server. An additional PII masking mechanism is applied during the parsing pipeline.

Administrators define graduation categories with flexible, department-scoped names rather than being limited to a fixed set of labels. The system supports department-based category pools, cohort enrollment bounds, curriculum equivalence rules, conditional thresholds, prefix sub-limits, and global department-level checks (such as minimum total ECTS).

To handle high user traffic during graduation periods, the application is developed using a microservices architecture with Dockerized services and Kubernetes for orchestration.

# Positioning

## Problem Statement

| The problem of | The manual evaluation of university students' graduation statuses (In-Department/Out-of-Department Mandatory, Technical/Non-Technical Elective courses, and credit/ECTS calculations) based on transcripts, and the high margin of error associated with this process |
| --- | --- |
| affects | Students, Teachers, and Department Administrative Staff / Student Affairs |
| the impact of which is | The slowdown of verification processes due to heavy workload during graduation periods, students experiencing grievances resulting from human errors, significant time lost by academic advisors, and the inability to immediately and transparently view the reasons for failing to meet graduation requirements |
| a successful solution would be | The deployment of a centralized system capable of dynamically managing graduation requirements, automatically parsing uploaded transcript PDFs, protecting personal data, and instantly generating definitive results |

## Product Position Statement

| For | Students, Teachers, and University administration |
| --- | --- |
| Who | Need to manage and track graduation verification processes quickly, accurately, transparently, and with respect for personal data |

| That | Automatically processes PDF transcripts according to the up-to-date course categories, codes, and credit/ECTS rule sets defined by Admins, presenting the definitive graduation status and the reasons for any deficiencies within seconds |
| Unlike | Traditional manual verification methods and manual tracking systems |
| Our product | Provides an uninterrupted experience even during sudden traffic spikes, thanks to high flexibility, speed, and ease of use. It ensures data privacy by processing student data instantly without storing it on the server |

## User Environment

TAMS is designed to integrate into the daily academic and administrative processes of different user profiles within the university ecosystem.

**Access and Platform:** The system is a fully web-based application. Users access the system from anywhere with an internet connection via modern web browsers (Chrome, Safari, Firefox, Edge, etc.) without requiring any client installation.

**Device Usage:**

- **Admins and Teachers:** Primarily perform operational tasks such as defining rule sets, creating course categories, and uploading PDF transcripts in an office environment, using desktop or laptop computers.
- **Students:** Since they only have viewing privileges, they heavily use mobile devices (smartphones and tablets) to access the system. The student interface (`/student/results`) must therefore be fully responsive and mobile-friendly.

**Task Cycle Time:** The core analysis process is designed to be near-instantaneous. A teacher uploads a transcript PDF; the system parses it asynchronously via a message pipeline and displays the result on screen within seconds — eliminating the minutes or hours spent on manual verification.

**Usage Spikes:** System usage is tied to the academic calendar. While usage remains at low-to-medium levels during mid-semesters, there are sudden and extremely high spikes in concurrent users at the end of semesters, and particularly during graduation weeks.

**Relationship with Other Applications and Integration:** In the initial phase (Release 1 / MVP), the system operates independently from the university's other information systems, utilizing its own authentication infrastructure (JWT-based login). The system architecture is structured to support integration with the university's existing Central Authentication services (LDAP / Active Directory, etc.) in Release 2.

# Product Overview

## Needs and Features

| Need | Priority | Features | Planned Release |
| --- | --- | --- | --- |
| Secure Access and Authentication | Must | Login and logout for Admins, Teachers, and Students using independently assigned accounts. JWT-based authentication, role-based route protection, and mandatory password change on first login | Release 1 (MVP) |
| Management of Graduation Rules | Must | Admins add, update, delete, and view departments, the global course catalog, graduation categories (with course codes, credits, ECTS, and mandatory flags), and curriculum equivalence rules | Release 1 |
| Automated Transcript Analysis | Must | Teachers upload PDF transcripts; the system parses the PDF asynchronously and compares extracted course data against Admin-defined rules within seconds | Release 1 |
| Transparent Result Display | Must | Teachers and Students (if their transcript is uploaded) view graduation status; if ineligible, they see a detailed list of missing credits and required courses per category | Release 1 |
| Student History Tracking | Should | Teachers view graduation data of previously analyzed students in a paginated, searchable list (`/teacher/history`) | Release 1 |
| Protection of Personal Data | Must | PDF documents are not permanently stored on the server; operations execute in-memory before the document is destroyed. PII masking is applied during parsing. Kafka topic retention limits exposure window | Release 1 |
| Integration with Institutional Systems | Could | Users log in with institutional credentials by connecting the system to the university's existing authentication infrastructure (LDAP / Active Directory) | Release 2 |

# Other Product Requirements

| Requirement | Priority | Planned Release |
| --- | --- | --- |
| The system must provide uninterrupted service during sudden traffic spikes, especially at the end of semesters and during graduation weeks. Auto-scaling infrastructure on Kubernetes will be utilized. The asynchronous data flow between the main backend service and the parser service will be handled via Kafka in KRaft mode | Must | Release 1 |
| All communication between the client and the server must be encrypted using HTTPS/TLS. Student transcripts (PDFs) must under no circumstances be written to physical server disks; they must be processed entirely in temporary memory and destroyed immediately | Must | Release 1 |
| The system will be designed with a microservices architecture, and all components will be packaged as Docker containers. The main backend services will be developed with Java (Spring Boot), the user interface (Frontend) with React, and the PDF Parser microservice with Python | Must | Release 1 |
| Considering that students will predominantly access the system from mobile devices, the frontend architecture of the application (particularly the student result screens) must be designed to be fully responsive | Must | Release 1 |
| A User Manual explaining step-by-step operations must be prepared for Admins who input the rules and Teachers who upload PDFs | Should | Release 1 (TBD) |
| Swagger/OpenAPI documentation detailing the API endpoints of the microservices must be provided for the development and integration process | Should | Release 1 |
